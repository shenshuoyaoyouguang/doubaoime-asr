# Agent Notes

- 当前开发机上运行 `pytest` 时，默认 `%LOCALAPPDATA%\\Temp\\pytest-of-76578` 可能触发 `PermissionError: [WinError 5]`。
  - 建议在执行测试前显式设置工作区内临时目录，例如：
    - PowerShell：`$env:TMP = (Resolve-Path .tmp-pytest).Path; $env:TEMP = $env:TMP`
  - 否则 `tmp_path` 相关测试会误报失败，掩盖真实结果。
- `doubaoime_asr/agent/stable_simple_app.py` 的 compat 包装层不能直接覆盖运行态实现。
  - `SessionManager._on_event` 必须桥接回 coordinator 事件队列，否则 `run()` 路径会丢 worker interim/final/finished/exit 事件。
  - `InjectionService.begin_session()` 创建的运行态 inline composition 状态不在 compat `app._session` 上；inline/最终提交逻辑必须保留 runtime 原始实现，只有显式测试会话覆盖时才走 compat 包装。
  - 覆盖协调器方法时要先保存原始实现；像 `_send_stop_if_needed()` 这类互相委托的方法如果双向重定向，容易产生递归。
  - wrapper 级 `is_current_process_elevated` / `restart_as_admin` monkeypatch 需要在 coordinator 构造后立即同步到 coordinator/injection_service，不能只在 coordinator 构造函数里计算一次。
  - 旧 `StableVoiceInputApp._handle_press()` API 仍可能被直接调用；当 `_ensure_worker()` 返回运行态 `_SessionCompatWrapper` 时，必须走 `session_manager.begin_session()` + `injection_service.begin_session()`，不能继续把 `composition=` / `inline_streaming_enabled=` 转发给运行态 `session_manager.WorkerSession.begin()`。
  - 旧 `StableVoiceInputApp._handle_worker_event(session_id, event)` API 仍需要保留 `session_id` 守卫；过时 worker 的 delayed finished/error 事件必须被丢弃，不能直接转发给 coordinator 覆盖当前活跃会话。
- `doubaoime_asr/agent/coordinator.py` 本体也必须自带运行态连线：
  - 直接实例化 `VoiceInputCoordinator` 时，`SessionManager` 需要在构造时就绑定 `on_event`，否则 worker stdout 解析出的 ready/interim/final/finished 事件不会进入协调器事件队列。
  - `VoiceInputCoordinator` 在启动时计算出的 `_process_elevated` 需要同步写入 `InjectionService.set_process_elevated()`；否则管理员目标仍会被 `target_requires_admin()` 误判为不可注入。
- Windows 上跑 `tests/test_stable_simple_app.py` / `tests/test_coordinator.py` 相关用例时，当前仓库可能出现 `PytestUnraisableExceptionWarning`，栈里是 `BaseSubprocessTransport.__del__` / `Event loop is closed`。
  - 这看起来是已有的 asyncio subprocess 清理时序问题，不是本轮 interim dispatcher / inline profile 改动直接引入的失败。
  - 如果后续有人继续清理这块，请优先排查 `SessionManager`/worker 相关测试 double 关闭 transport 或在 loop 关闭后析构 subprocess pipe 的路径。
- 全量 `pytest -q` 的 warning 数并不稳定；在 2026-04-07 的一次 Batch 1 重构后，新增测试会把同一类历史问题放大成 **2 条** warning，集中在 `tests/test_hotkey_service.py::TestHotkeyServiceUpdateHotkey::test_update_hotkey_while_running`。
  - 两条 warning 分别来自 `_ProactorBasePipeTransport.__del__` 和 `BaseSubprocessTransport.__del__`，都指向已关闭 pipe 的 `fileno()` / transport `__repr__`。
  - 这更像是既有 hotkey/asyncio 子进程测试清理问题被不同测试排序触发，而不是 ASR / settings / transcript 拆分直接引入的新业务回归。
- 2026-04-08 进一步定位后确认：`tests/test_coordinator.py::TestCoordinatorConfigChanges::test_apply_config_invalidates_preflight_on_credential_path_change` 之前没有 mock `session_manager.restart_worker()`，会在单测里**真的拉起 worker subprocess**。
  - 该 subprocess 泄漏是全量 `pytest -q` 剩余 `BaseSubprocessTransport.__del__` warning 的直接来源之一。
  - 修复方式：在该测试中显式 `coordinator.session_manager.restart_worker = AsyncMock()`，把它固定成纯单测。
  - 修复后全量回归恢复为 **433 passed, 0 warnings**。

  - 这说明 **warning 的触发点在 hotkey 测试文件，根因大概率不在 hotkey 写集本身**；更像是别处泄漏的 asyncio subprocess transport 在该文件执行期间才被 GC/析构。
  - 后续若要继续清零 warning，优先排查 `session_manager` / `stable_simple_app` / 其他 subprocess-heavy 路径，而不是继续深挖 `hotkey_service`。
